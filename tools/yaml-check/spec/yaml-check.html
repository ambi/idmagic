Glossary 10

曖昧語、別名、翻訳、外部標準語を説明する補助用語集。

YamlInput

検査対象となる UTF-8 テキストファイル。拡張子は通常 .yaml だが、ツール側は内容ベースで処理する。

Finding

1 つの検査違反を表すレコード。発生したファイル・行・列・メッセージを持つ。 Parse / Lint / Schema のいずれかの層で生じる。

aliases —
ParseFinding

Bun の YAML ローダで構文解析に失敗したことを示す Finding。

aliases —
LintFinding

生テキストへの形式的なチェック (tab indent / 行末空白 / 末尾改行) に 違反した Finding。意味解釈は行わない。

aliases —
SchemaFinding

Parse 成功後のデータが、明示指定された Schema に適合しなかったことを 示す Finding。原因の JSON pointer と Ajv のメッセージを内包する。

aliases —
ContextMapFinding

context_map セクションの意味的整合性 (参照解決・非循環・shared_kernel の 肥大) に対する Finding。Schema が検査できない context 間の関係を見る。 severity が error のものは exit code に反映し、warning のものは反映しない。

aliases —
Severity

Finding の重大度。error は CheckRun の失敗を意味し exit code 1 に反映する。 warning は助言であり exit code には反映しない。Parse / Lint / Schema 由来の Finding は常に error。ContextMapFinding のみ warning を取りうる。

aliases —
Schema

JSON Schema 2020-12 形式の検査定義。本ツールに同梱され、`SchemaName` の 列挙値で呼び出される。

aliases json-schema
SchemaName

同梱 Schema の識別名。`work-item` / `scl` の 2 種。

aliases —
CheckRun

1 回の `yaml-check` 起動。引数で指定されたファイル群に対して順に Parse → Lint → (任意) Schema を回す。

Invariants 12

入力に依らず常に成り立つ不変条件、または決して起きてはならない事象。

parse_before_lint_and_schema

must

各ファイルに対し parse が最初に走る。parse 失敗時は schema 検証を行わない (lint は生テキスト依存なので、parse 失敗とは独立に実行する)。

target
CheckYaml
always (invariant)
for every file f: parse(f) is evaluated before schema(f), and schema(f) only runs when parse(f) succeeded

schema_is_opt_in

must

入力ファイル名やパスから schema を推測してはならない。schema は CLI で明示指定された場合にだけ実行される。

target
CheckYaml
never (forbidden)
schema validation runs for a file when --schema was not provided

schema_name_validated_upfront

must

--schema に未知の名前が渡されたら、ファイル検査を一切行わず exit code 2 で 終了する。誤指定で部分結果を返さない。

target
CheckYaml
never (forbidden)
any file is checked when --schema=<name> and name is not in SchemaName

schema_application_is_uniform

must

--schema=<name> を指定すると、入力ファイル全件に同じ schema を適用する。

target
CheckYaml
always (invariant)
every input file is validated against the same schema in a single CheckRun

no_arbitrary_code_execution

must

入力 YAML の内容によって、ツール側で任意コードが実行されてはならない。 anchor / tag / merge key 等の高度な YAML 機能による副作用は許可しない。

target
CheckYaml
never (forbidden)
evaluating CheckYaml causes execution of code originating from YamlInput contents

no_input_modification

must

yaml-check は読み取り専用である。入力ファイルへの書き戻し・自動修正は行わない。

target
CheckYaml
never (forbidden)
CheckYaml mutates any file under the file system

deterministic_findings

must

同じ入力ファイル群と同じ --schema 引数からは、findings 集合とその順序が 常に同一になる。時刻・乱数・環境変数に依存しない。

target
CheckYaml
always (invariant)
CheckYaml(files, schema) ≡ CheckYaml(files, schema)

exit_code_reflects_findings

must

CheckRun が 1 つでも error 重大度の Finding を出した場合 exit code は 1。 error が 0 件なら 0 (warning のみは 0)。引数自体の誤りは 2。3 つの状態が 混じってはならない。

target
CheckYaml
always (invariant)
exit_code(CheckRun) = 0 iff no error-severity Finding, 1 iff any error-severity Finding, 2 iff usage_error

context_map_checked_when_present

must

入力が context_map セクションを持つとき、--schema の有無に関わらず 参照解決・非循環の検査が走る。未解決参照と循環は error、shared_kernel の 肥大は warning とする。

target
CheckYaml
always (invariant)
for every input with a context_map: unresolved depends_on/uses and dependency cycles are reported as error-severity ContextMapFinding

work_item_dependencies_are_resolved_and_acyclic

must

workspace の未完了 work-item が宣言する depends_on は、同じ work-item 名前空間の open / done を含む既知の WI を指す。自己参照と循環依存は error として報告する。

target
CheckYaml
always (invariant)
every workspace work-item dependency refers to an existing non-self work item, and the dependency graph is acyclic

warnings_do_not_fail_the_run

must

warning 重大度の Finding は表示するが CheckRun の成否を変えない。

target
CheckYaml
never (forbidden)
a CheckRun with only warning-severity Findings exits non-zero

schema_finding_points_into_source

must

SchemaFinding は、Ajv の instancePath を入力テキスト上の行番号に解決して 添付する。解決に失敗しても列 1 / 行 1 を返し、エラー化しない。

target
CheckYaml
always (invariant)
every SchemaFinding carries a (line, column) within bounds of the source file

Scenarios 9

受け入れ例。自然文ステップで書かれた振る舞いの具体例。引用された既知の名前は定義へリンクする。

正しい work-item を --schema=work-item で検査すると ok になる

Steps
  1. 規定どおりの work-item.md を用意する
  2. --schema=work-item を指定して yaml-check で検査する
  3. すべてのファイルが ok と報告される
  4. exit code は 0

Plan と Tasks を持つ work-item は --schema=work-item で検査できる

Steps
  1. # Plan と T001 形式の checkbox を含む # Tasks を持つ work-item.md を用意する
  2. --schema=work-item を指定して yaml-check で検査する
  3. schema 由来の Finding は出ない
  4. exit code は 0

work-item の完了前提は workspace 全体で検査される

Steps
  1. pending work-item に既存 WI ID を depends_on として列挙する
  2. workspace の work-item 検査を実行する
  3. 参照先が存在し、自己参照も循環もなければ成功する
  4. 未知の ID、自己参照、または循環があれば該当 WI の error Finding と exit code 1 を返す

未知のスキーマ名を渡すと、ファイル検査は行わず exit code 2 を返す

Steps
  1. --schema=bogus と任意のファイルを渡して yaml-check を呼ぶ
  2. エラーメッセージに利用可能な SchemaName 一覧が含まれる
  3. 検査対象ファイルは 1 件も読み込まれない
  4. exit code は 2

--schema を付けずに呼ぶと parse + lint のみが走る

Steps
  1. スキーマ違反だが YAML として valid なファイルを用意する
  2. --schema を付けずに yaml-check で検査する
  3. parse 由来 / lint 由来の Finding は出るが schema 由来の Finding は出ない

parse 失敗ファイルでは schema 検証はスキップされる

Steps
  1. YAML として破損した内容を持つファイルを用意する
  2. --schema=scl を指定して yaml-check で検査する
  3. parse の Finding が報告される
  4. schema 由来の Finding は同じファイルに対し追加で出ない

--list-schemas は同梱スキーマ名のみを出力する

Steps
  1. --list-schemas を指定して yaml-check を呼ぶ
  2. 標準出力に work-item / scl が 1 行ずつ書かれる
  3. exit code は 0

context_map の未解決参照や循環は error として検出される

Steps
  1. depends_on が存在しない context を指す、または publish されていない名前を uses する context_map を用意する
  2. yaml-check で検査する
  3. ContextMapFinding (error) が報告される
  4. exit code は 1

shared_kernel の肥大は warning であり exit code を変えない

Steps
  1. via が shared_kernel で多数の名前を共有する context_map を用意する
  2. yaml-check で検査する
  3. shared_kernel の warning が表示される
  4. 他に error が無ければ exit code は 0
