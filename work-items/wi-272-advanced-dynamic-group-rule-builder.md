# 動的グループルールビルダーの作成画面への拡張と、任意CEL式のビジュアル編集（逆パース）対応

## Motivation
wi-216 で動的グループの編集画面に簡易的な条件ビルダーUI（AND条件のみをサポート）を追加したが、以下の課題が残っている。
1. **グループ作成画面でビルダーが使えない**: 新規作成時は手書きでCEL式を書く必要があり、直感的ではない。
2. **既存式の逆変換（パース）不可**: 手書きで作成した式や、ORを含むような式はビルダーで再編集できず、生CELの直接編集にフォールバックしてしまう。
3. **表現力の限界**: 現在のビルダーは単純な `属性 演算子 値` の AND 結合しか表現できず、ネストされた条件や関数（例: `has()`, `exists()`）などの高度なCEL式を視覚的に表現できない。

これらを解決し、どのようなCEL式でもビジュアルに編集・組み立てができ、かつグループ作成時にも利用可能にするための改善を行う。

## Scope
- **ui**:
  - `frontend/src/features/admin-groups/AdminGroupCreatePage.tsx` への `DynamicRuleEditor` コンポーネントの導入。
  - `DynamicRuleEditor` および関連ロジックの刷新。単なる AND 条件の配列から、CEL AST（抽象構文木）を表現できるツリー型またはブロック型のビジュアルエディタへのアップグレード。
  - バックエンド、あるいはフロントエンドのパーサを用いた、生CEL文字列からビジュアルエディタ状態への逆パース処理の実装。
- **backend** (必要に応じて):
  - CEL文字列をパースして AST JSON を返すエンドポイントの追加（フロントエンドでJS製パーサを用いない場合）。

## Out of Scope
- 動的ルールの評価エンジン自体の変更（既存のCEL評価器はそのまま使用）。
- CEL仕様外の独自DSLの導入。

## Plan
- **作成画面への導入 (Low Risk)**: `AdminGroupCreatePage.tsx` の状態管理を調整し、新規作成フローにも `DynamicRuleEditor` を埋め込む。
- **任意のCEL式の表現・編集 (High Risk)**: 任意の式（OR、ネストされた括弧、マクロ、関数呼び出し等）をビジュアルに表現するには、AST（抽象構文木）をベースとしたツリー状のエディタ（例: 条件グループのネストが可能なUI）が必要になる。要件を満たす表現力と、非エンジニアにも使いやすいUIのバランスを設計する。
- **CELからUI状態への逆パース (High Risk)**: 入力されたCEL文字列からASTを構築し、UI状態に復元する。フロントエンド側でCELの字句解析・構文解析を行うライブラリを導入するか、バックエンドの `cel-go` を利用してパース結果（AST表現）をAPI経由で取得する方式のいずれかを採用する。

## Verification

- 手動: `(user.department == "Engineering" || user.department == "Sales") && user.email_verified == true` のような複雑な式を手書きで入力し、ビルダーUIに正確にパース・展開されることを確認。
- 手動: 展開されたビルダーUI上で条件を編集し、正しいCEL文字列が再生成されることを確認。

## Risk Notes
- **実装コストと複雑性の増大 (High)**: 任意のCEL式を表現できるビジュアルエディタの開発は、実質的にビジュアルプログラミング言語（ノードベースエディタ等）の開発に近くなり、フロントエンドの複雑度が跳ね上がるリスクがある。
- **パーサの維持管理**: CELの構文解析をフロントエンドで行う場合、バックエンド（`cel-go`）との挙動の一致を保証する必要がある。バックエンドにパース用エンドポイントを設ける方が堅牢だが、APIコールのレイテンシがUXに影響する可能性がある。
