# Change Record Format

Regenerative Architecture が要求する**変更管理レコード**——ワークアイテムと決定記録
（ADR）——の正本フォーマットを定義する。ここに書かれた書式がマスターであり、機械検証用の
スキーマ（既定では JSON Schema）はこの文書からの派生物として扱う。

`REGENERATIVE_ARCHITECTURE.md` はこれらのレコードを概念として規定し（決定記録は §3.2、
ワークアイテムは §4.2 / §4.2.1）、本文書はその**記法**を定める。SCL に対する
`SPECIFICATION_CORE_LANGUAGE.md` と同じ役割を、変更管理レコードに対して担う。別プロジェクトで
別の書式を採るときは、本文書だけを差し替える。

新規にレコードを書くときは、**既存ファイルを開いて書式を確認しない**。本文書の書式に従う。
既存ファイルは「似た題材の中身」を参照したいときだけ開く。

見出し名は記法上の固定要素である。本文書は日本語の見出しを正本とし、他言語プロジェクトは
構造を保ったまま翻訳してよい。

---

## 1 ワークアイテム

一つの意味変更として説明・実装・検証できる作業単位。正本は、課題管理サービスに依存しない
スキーマ検証可能なテキスト形式（Frontmatter付きMarkdown）で、対象となる境界づけられたコンテキストの
近くに保存する。

### 1.1 配置と命名

- ファイル名は `work-items/<id>.md`。`<id>` は `<接頭辞>-wi-<連番>-<kebab-title>`。
  `<接頭辞>` はレコードを所有するコンテキストを表す短い固定コード（1.1.1 の対応表）、
  `<連番>` はそのコンテキスト内でのみ採る連番。接頭辞で名前空間が分かれるので、`id` だけで
  所有コンテキストが判別でき、別コンテキストの同一連番と混同・衝突しない
  （例 `idp-wi-23-...` と `repo-wi-23-...` は別物として並立する）。
- 特定コンテキストの作業はそのコンテキスト配下の `work-items/` に置き、リポジトリ全体の規約・
  横断ツール・複数コンテキストにまたがる作業だけをルートの `work-items/`（接頭辞 `repo`）に置く。
- **未着手・進行中（`pending` / `in_progress`）は `work-items/` 直下に置く。**
  **完了・中止（`completed` / `cancelled`）は同じ `work-items/` 下の `done/` サブディレクトリに移す**
  （例 `work-items/done/<id>.md`）。これで「まだ動きのある作業」と「終わった作業」を、
  ファイルを開いて `status` を読まなくても配置だけで区別できる。`status` は引き続き正本であり、
  `done/` への配置はその冗長な物理表現にすぎない。`<id>` とファイル名は移動後も変えない。

### 1.1.1 コンテキスト接頭辞

各コンテキストは短い接頭辞コードを 1 つ持ち、その `work-items/` と `decisions/`（2.1）の
両方で使う。コードとディレクトリの対応はこの表で一元管理し、新しいコンテキストを足すときは
ここに 1 行加える。接頭辞はディレクトリから機械的に導けるので、ファイルの置き場所を見れば
`id` の先頭が定まる。

| コンテキスト | ディレクトリ | 接頭辞 |
| --- | --- | --- |
| リポジトリ全体（手法・横断ツール・複数コンテキストにまたがる作業） | ルートの `work-items/` `decisions/` | `repo` |
| アプリ実装（例：`<app>` という境界を足す場合） | `<app>/work-items/` `<app>/decisions/` | `<app>` |

### 1.1.2 採番と衝突回避

- 連番はコンテキスト内の既存最大値 + 1。接頭辞で名前空間が分かれるためコンテキストをまたぐ
  衝突は起きないが、**同一コンテキスト**で別ブランチ・別作業者が並行して採番すると、同じ
  連番を採ってしまうことがある（それぞれのブランチでは既存最大値が同じに見えるため）。
- 連番そのものは正本ではなく、この衝突は**防ぐのではなく検出して直す**。`tools/` の重複 `id`
  検出チェックを pre-commit / CI で走らせ、同一ディレクトリ内に重複 `id` があればマージを
  落とす。被ったら後発のレコードの連番を採り直すだけでよい。相互参照が付く前（マージ前）に
  直すので、影響は起票直後のレコードに閉じる。
- いったんマージされ他所から参照され始めた `id` は変えない。連番は再利用しない
  （done への移動で `<id>` を変えない 1.1、廃止 ADR を残す 2.1 と同じ方針）。

### 1.2 フィールド（Frontmatterメタデータ）

| フィールド | 必須 | 内容 |
| --- | --- | --- |
| `id` | ✓ | ファイル名と一致する安定識別子。`^[a-z0-9][a-z0-9-]*$` |
| `title` | ✓ | 一文で表す意味変更 |
| `created_at` | ✓ | 起票日。`YYYY-MM-DD` または RFC3339 |
| `authors` | ✓ | 1 名以上 |
| `status` | ✓ | `pending` \| `in_progress` \| `completed` \| `cancelled` |
| `risk` | ✓ | `low` \| `medium` \| `high` \| `critical` |

### 1.3 本文（Markdownセクション）

Frontmatter の直後に `title` を H1（`#`）として 1 つ置き、各セクションは H2（`##`）で書く。
見出しレベル 1 はファイルに 1 つだけに保つ（markdown プレビューや linter の慣習に沿う）。

| セクション見出し | 必須 | 内容 |
| --- | --- | --- |
| `## Motivation` | ✓ | なぜこの変更が必要か（What ではなく Why）。 |
| `## Scope` | ✓ | 対象範囲。触れる予定の SCL セクションやコードモジュールを含める。 |
| `## Out of Scope` | ✓ | 明示的にやらないこと。 |
| `## Plan` | △ | 中規模以上の work-item で必須。技術方針、触れる層、主要な設計選択、却下した代替案、人間承認が必要な未決定事項。 |
| `## Tasks` | △ | 中規模以上の work-item で必須。途中再開できる粒度の checkbox。`in_progress` では進捗の正本。 |
| `## Verification` | ✓ | 予定する検証コマンド / 手動手順。 |
| `## Risk Notes` | ✓ | リスクの根拠と軽減策。 |
| `## Completion` | △ | `status` を `completed` / `cancelled` にする時点で必須。完了日時、要約、実施された検証結果などを箇条書きで追記する。 |

`## Plan` は ADR の代替ではない。ワークアイテム内での実装方針を固定する短命の設計契約であり、
他ワークアイテムにも効く決定、後から再生成に必要な決定、廃止や置換を追跡すべき決定は ADR に
昇格する。Plan は、より根底にある Architecture（`ARCHITECTURE_FORMAT.md`）が定める採用スタック・
依存の向き・構造に準拠して書く。構成そのものを変える方針は Plan に留めず Architecture に反映する。

`# Tasks` の推奨形式は次の通り。

```markdown
- [ ] T001 [SCL] scenario と invariant を更新する。
- [ ] T002 [App] Use case を実装する。
- [ ] T003 [Adapter] HTTP handler と contract test を追加する。
- [ ] T004 [Verify] `just verify` を実行する。
```

`[SCL]` / `[ADR]` / `[App]` / `[Adapter]` / `[Runtime]` / `[UI]` / `[Docs]` /
`[Verify]` のように、どの層または検証境界の作業かを明示する。並列可能なタスクは説明中に
`[P]` を付けてよい。各タスクは、原則として 1 つの層または 1 つの検証境界に閉じる。

次のいずれかに該当するワークアイテムは、中規模以上として `## Plan` と `## Tasks` を書く。

- 複数のシナリオ、ユーザーストーリー、または独立した利用者価値を含む
- RA の 3 層以上にまたがる
- DB migration、認可、外部契約、破壊的変更、不可逆な移行を含む
- 予定する作業が 1 セッションで終わる確信を持てない
- 検証が複数サブシステムにまたがる

### 1.4 スケルトン

```markdown
---
id: idp-wi-NN-kebab-title   # <接頭辞>-wi-<連番>-<kebab-title>。ファイル名と一致する安定識別子
title: "一文で表す意味変更"
created_at: 2026-01-01      # YYYY-MM-DD
authors: [name]
status: pending             # pending | in_progress | completed | cancelled
risk: low                   # low | medium | high | critical
---

# 一文で表す意味変更

## Motivation
なぜこの変更が必要か（What ではなく Why）の背景。

## Scope
- `spec/scl.yaml` の `interfaces.StartTask`
- `src/usecase/` への実装

## Out of Scope
- 明示的にやらないこと。

## Plan
- 採る技術方針、触れる層、却下した代替案、未決定事項。

## Tasks
- [ ] T001 [SCL] 仕様を更新する。
- [ ] T002 [App] 実装する。
- [ ] T003 [Verify] 検証する。

## Verification
- 予定する検証コマンド（例：`go test ./...`）や手動手順。

## Risk Notes
リスクの根拠と軽減方法。

## Completion
- **Completed At**: 2026-01-01
- **Summary**:
  実装した変更の意味上の差分の要約。
- **Verification Results**:
  - `go test ./...` - passed
```

先頭の H1 は frontmatter の `title` と同じ一文でよい。

## 2 決定記録（ADR）

重要な決定とその理由を残すレコード。SCL から派生し SCL に反映される。配置はワークアイテムと
同じく対象コンテキストの近くの `decisions/` に置く。

### 2.1 配置と採番

- ファイル名は `decisions/<接頭辞>-ADR-NNN-kebab-title.md`。`<接頭辞>` は 1.1.1 の対応表と
  同じコンテキストコード、`NNN` はそのコンテキスト内の 3 桁ゼロ詰め連番
  （例 `idp-ADR-024-...`、ルートの手法レベルなら `repo-ADR-001-...`）。接頭辞で名前空間が
  分かれるので、`id` だけで所有コンテキストが判別でき、相互参照（`idp-ADR-015 に置き換え`）も
  一意に定まる。
- 連番はコンテキスト内でのみ採り、再利用しない。廃止した ADR も**削除せず**残す
  （過去の決定経緯は再生成の文脈になる）。
- 並行採番による同一 `decisions/` 内の連番衝突は、ワークアイテムと同じく検出して直す（1.1.2）。

### 2.2 章構成

次の見出しをこの順で持つ。

1. **ステータス** — `採用` / `提案` / `廃止`。廃止時は置き換え先 ADR と日付を添える
   （例: `廃止（idp-ADR-015 に置き換えられた、2026-03-01）`）。決定が SCL に反映されているときは、
   ここまたは「影響」で対応する SCL 要素とソースを相互参照する。
2. **コンテキスト** — なぜ今この決定が要るか。制約・力学・先行する決定との関係。
3. **決定** — 何を決めたか。番号付きで複数項目に分けてよい。
4. **却下した代替案** — 各案と、それを採らない理由。
5. **影響** — 変更される SCL 要素・契約・データ・運用など。

「決定」と「却下した代替案」の間に、決定を補足する任意のトピック節を置いてよい
（例: `## 鍵 of サイズと曲線`）。トピック節は決定の一部であり、必須章を置き換えない。

### 2.3 スケルトン

```markdown
# idp-ADR-NNN: 一文で表す決定

## ステータス
採用。`scl.yaml` の `interfaces.Xxx` / `models.Yyy` と該当ソースに反映。

## コンテキスト
なぜ今この決定が要るか。

## 決定
何を決めたか。

## 却下した代替案
- 案 A: なぜ採らないか。

## 影響
- 新規 / 変更される SCL 要素、契約、データ、運用。
```

---

機能変更で**どの SCL セクションを見直すか**は、SCL 側の関心であり
`SPECIFICATION_CORE_LANGUAGE.md` §3 冒頭の網羅チェックに従う（SCL-first）。
